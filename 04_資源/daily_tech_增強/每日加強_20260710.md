## 今日主題：系統架構設計與資料庫優化（星期五）

### 深入學習：分庫分表策略——水平擴充的最佳實踐 📊

當單一資料庫的寫入吞吐量開始飽和，垂直切分（加更多 CPU/記憶體）已無法滿足需求時，**分庫分表（Sharding）** 成為最常見的水平擴展方案。其核心思想很簡單：把一張大表拆成多張小表，分散到多台資料庫上，讓每台的負載降低 N 倍（N = 機器數量）。

#### 兩種切分方式

1. **水平切分（分表）**：同一張大表分成多張結構相同的小表。例如 users -> users_0, users_1, ..., users_31。適合按 ID、時間等字段做 hash 或 range 分配，讀寫均衡分散。
2. **垂直切分（分庫）**：按業務劃分不同的資料庫。例如把「使用者服務」和「訂單服務」放到不同 DB。優點是各業務可獨立擴展、降低耦合；缺點是跨庫關聯查詢需要額外的協調成本。

#### 路由策略：怎麼決定資料放哪台？

常見的 Three Strategies，各有取捨：
- **Hash（均勻性最佳）**：對分片鍵做 hash（如 id % 16），寫入數據均勻分散，但新增機器時 redistribution 範圍大。
- **Range / Range Partition（時間字段經典）**：按年月拆表。利於時間 range 查詢和資料淘汰，但热点月份的問題要小心處理。
- **Lookup（解耦鍵值）**：把路由鍵做二次查表決定目標，新增/刪除分片時影響面最小。缺點是多一次查詢成本。

#### 常見陷阱（實戰經驗提醒 ⚠️）
- **跨庫 JOIN**：避免在應用層組合多張表的 SQL；可考慮用存儲過程或寬表預聚合來解決。
- **索引維護**：每張分片表都有獨立索引，查詢前先決定路由鍵，才能命中局部索引。隨機條件查詢會退化為全庫掃描（cost = N 倍）。
- **資料遷移**：線上分片的同時必須對寫入進行「雙寫 + 補齊」策略，確保數據零丟失。
