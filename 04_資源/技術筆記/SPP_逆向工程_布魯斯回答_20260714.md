## 1. SPP 是什麼（確認理解）

SPP = **Bluetooth Serial Port Profile（藍牙串列埠協議）**。它在藍牙協議棧上模擬一條 RS-232 串口線，讓兩個藍牙設備間可以像接了串口一樣收發**原始位元組流**。

- Android 上通過 `BluetoothDevice.createRfcommSocketToServiceRecord(UUID)` 建立 `BluetoothSocket`，走 `getInputStream()/getOutputStream()` 收發。
- UUID 通常是標準 SPP UUID `00001101-0000-1000-8000-00805F9B34FB`，或廠商定義的私有 UUID。
- 常見於 IoT / 智能硬體配套 App：藍牙模組（HC-05/HC-06）、醫療設備、車機、藍牙印表機、電子鎖、遙控設備等。
- 協議內容通常是**廠商定義的私有指令幀**（幀頭/幀尾/長度/命令碼/校驗/加密），由 App 組幀下發、解析回傳。

✅ 理解與勞大說的一致，就是「硬體配套 App 透過藍牙串口跟設備講私有協議」這回事。

## 2. 技術可行性與工具

**可行。** 但關鍵認知：SPP 是本地藍牙串行，**不走網路**，所以 BurpSuite 抓不到它（除非 App 另有雲端 API）。逆向重點在兩處：
1. 靜態反編譯找回指令協議（幀格式 / 校驗 / 加密）。
2. 運行時 Hook `BluetoothSocket` 的讀寫，dump 實際收發位元組。

工具分層：

| 類型 | 工具 | 用途 |
|---|---|---|
| 靜態 | **jadx-gui** | dex→Java 源碼，讀邏輯 |
| 靜態 | **APKTool** | 反編譯資源 / Manifest / 重打包插樁 |
| 動態 | **Frida** | Hook `BluetoothSocket` 讀寫、加密/校驗函數 |
| 動態 | **Objection** | Frida 封裝，免寫腳本快速探索 |
| 抓包 | nRF Sniffer / Ubertooth | 空中抓包（但 SPP 是傳統 BR/EDR 藍牙，抓包難，通常改走 App 層 Hook） |
| 輔助 | **BurpSuite** | 僅當 App 同時有 HTTP/HTTPS 雲端；純 SPP App 用不大 |
| Native | IDA / Ghidra | 看 `libxxx.so` 裡的加密/白盒密鑰 |

結論：核心三件套是 **jadx + APKTool + Frida/Objection**，BurpSuite 是配角。

## 3. 實際操作步驟概述

1. **取得 APK**：`adb pull` 目標機，或官方/apkmirror 下載；注意 debug/release 簽名差異。
2. **反編譯**：jadx-gui 開 APK → 看 `AndroidManifest.xml` 找 `BLUETOOTH`/`BLUETOOTH_CONNECT` 權限、搜尋 `createRfcommSocketToServiceRecord`、`UUID.fromString`、`BluetoothSocket`、`getOutputStream` 關鍵字，定位通訊模組。
3. **靜態協議分析**：在源碼找組幀邏輯——幀頭/幀尾/長度/命令碼、**校驗（CRC16/checksum/XOR/自定義）**、**加密（AES/XXTEA）**；重點讀 `write(byte[])` 前組裝處與 `read()` 後解析處。
4. **動態驗證**：Frida/Objection Hook `BluetoothSocket` 的 `write`/`read`，dump 實際位元組，與靜態推測交叉驗證；必要時 Hook 加密/校驗函數拿明文-密文對。
5. **重組**：依幀格式寫 Python SPP 客戶端（`pybluez`）或 Android 端 `BluetoothSocket`，重現「連線→配對→下發指令→解析回傳」，做到能獨立驅動硬體。
6. **文檔化**：輸出協議幀、命令表、狀態機，交付其他同事接手。

## 4. 難點與風險

- **混淆**：R8/ProGuard/DexGuard/梆梆 把類名壓成 a/b/c，需結合字串、控制流、動態 Hook 還原。
- **校驗與防篡改**：指令帶 CRC/簽名；可能校驗設備 MAC/序列號；重打包簽名不符會被拒（需 Frida 繞簽名校驗或用原簽名）。
- **加密**：密鑰常硬編在 `libxxx.so`（native 層）或白盒，需 IDA/Ghidra。
- **反調試/反 Hook**：檢測 Frida/root/ptrace，需 anti-anti-frida 腳本繞過。
- **⚠️ 法律/授權邊界**：逆向第三方 App 可能違反其 EULA / 著作權法；為「互通性」目的的逆向在部分司法管轄有免責空間但不保證。建議**只對龍族自有或已授權的 App 做**；若目標是第三方商業 App，先取得書面授權或法律意見。這點請勞大拍板。

## 6. 給勞大的建議

- **要不要做**：目標若是「龍族自有硬體配套 App」或「已授權設備」→ 強烈建議，技術完全可行、ROI 高。若是第三方商業 App → 先解決法律授權（見第 4 點）再動手。
- **怎麼做**：先 jadx 靜態摸協議骨架 → Frida 動態 dump 實際流量交叉驗證 → Python SPP 客戶端重組。優先準備一台可 root 的測試機。
- **預估工時**（中等複雜度、無重度混淆/native 加密）：
  - 環境補齊（裝工具 + root 機）：0.5–1 天
  - 靜態協議分析：1–2 天
  - 動態攔截驗證：1–2 天
  - 重組客戶端 + 文檔：1–2 天
  - **合計約 4–7 個工作天**；若有 DexGuard/native 加密/反調試則翻倍。
