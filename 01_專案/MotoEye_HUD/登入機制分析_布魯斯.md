# MotoEye（com.vidoar.motohud）登入機制分析報告

> 分析人：布魯斯（Bruce）｜工具：androguard 4.1.4（python3）＋ 真實 Dalvik 指令反編譯
> 樣本：/tmp/MotoEye.apk（versionName = V1.0.2604141705，91 MB，12,074 個 class）
> 結論一句話：**登入是「本地 SP 有無 token 字串」決定的門面，完全沒有伺服器驗證啟動流程，去掉很容易、風險很低。**

## 二、啟動流程與登入判斷（反編譯出的真實控制流）

### 2.1 入口：SplashActivity 是唯一 LAUNCHER
AndroidManifest 中：
```xml
<activity android:name="com.vidoar.motohud.view.SplashActivity" android:exported="true">
   <intent-filter>
      <action android:name="android.intent.action.MAIN"/>
      <category android:name="android.intent.category.LAUNCHER"/>
   </intent-filter>
</activity>
```
整包 APK 只有 `SplashActivity` 帶 `LAUNCHER`。這就是我們要動手術的地方。

### 2.2 SplashActivity.init() 的判斷邏輯（真實 smali → 偽代碼）
反編譯 `SplashActivity.init()` 得到的關鍵指令：
```
invoke-static  InfoDataHelper.getInstance(ctx)
invoke-virtual   InfoDataHelper.getAppToken()      // 讀本地 token
invoke-static  TextUtils.isEmpty(token)          // v0 = 是否為空
const-wide/16  v1, 1500                          // 1.5 秒延遲
if-nez v0, +014h                                // ★ 關鍵分支
   ... sendEmptyMessageDelayed(what=0, 1500ms)   // token 為「空」→ 送 msg=0
   goto  +7h
   ... sendEmptyMessageDelayed(what=1, 1500ms)   // token 非「空」→ 送 msg=1
```
等價於：
```java
if (TextUtils.isEmpty(InfoDataHelper.getInstance(this).getAppToken())) {
    mHandler.sendEmptyMessageDelayed(0, 1500);  // → 登入頁
} else {
    mHandler.sendEmptyMessageDelayed(1, 1500);  // → 主頁
}
```

`SplashActivity$1.handleMessage()` 確認 msg 的歸宿：
```
if (msg.what == 0)  access$100() → startLoginPage()   // 跳 LoginActivity + finish()
else /*what==1*/    access$000() → startMainPage()     // 跳 Main2Activity + finish()
```

### 2.3 token 從哪裡來、存哪裡
- `InfoDataHelper.getAppToken()` 反編譯：
  ```
  iget-object v0, sp
  const-string v1, "app_token"
  invoke-interface sp.getString("app_token", null)
  ```
  → **純本地 `SharedPreferences`，讀 key `app_token`。**
- 寫入點：`UserController$5.onCallback()`（登入成功回呼）呼叫 `InfoDataHelper.saveAppToken(...)`。
- 全 APK 唯一會 `const-class ... LoginActivity` 跳過去的路徑，就是 `SplashActivity.startLoginPage()`（xref 證實 `startLoginPage` 只被 `SplashActivity$1` 呼叫）。**其餘任何主頁/功能頁都不再檢查登入。**

## 三、本地驗證 vs 伺服器驗證？（這是能否拿掉的核心）

**判斷：啟動門面是「純本地判斷」，伺服器驗證（若有）只在「已登入後」的雲端功能層發生，不參與「能否進 App」。**

證據：
1. 啟動分流只看 `TextUtils.isEmpty(getAppToken())`，**完全沒有** `UserController.login()` / 網路呼叫 / 伺服器回應介入 `init()` 流程。
2. 主頁 `Main2Activity` / `MainFunctionActivity` / `ConnectActivity` 的方法清單中**完全沒有** `getAppToken` / `isLogin` / `startLoginPage` → 進入主頁後不會再被踢回登入。
3. 網路層：`UserController` 用自家 `FormRequestBuilder` + `BaseController.enqueuePost(...)`（底層是 **okhttp3**，無 retrofit/volley）。`getAppToken()` 有 **40 個呼叫者**，幾乎全是 `UserController` / `TalkbackController` / `UploadController` / `OtherController` 的雲端 API，以及 `VPushManager.sendLogin()`。也就是：**token 綁定的是「雲端同步（POI/帳號/隊伍/通話）/ 推送」這類功能，不是地圖或藍牙 HUD 通訊。**
4. 核心藍牙 `BleCommService` 只有 `onLoginEvent` / `onAutoLogoutEvent` / `onTokenRefreshEvent` 來「被動接收」登入狀態，基礎 BLE 通訊（搜尋 HUD、連線、下發指令）**與登入無關**。

### 結論矩陣
| 功能 | 是否依賴登入 token | 拿掉登入後 |
|---|---|---|
| App 啟動 / 進主頁 | 否（只看本地 SP 有無字串） | ✅ 正常 |
| 藍牙 HUD 搜尋/連線/通訊 | 否 | ✅ 正常 |
| 地圖 / 導航（高德 Amap） | 否 | ✅ 正常 |
| 雲端 POI 收藏、帳號資料、隊伍對講、回報、推送 | **是** | ⚠️ 這類會 401 / 回傳錯誤，但只是「功能受限」不會崩 |
| 登入/註冊/重設密碼 UI | — | 變成廢頁（可保留） |

## 四、拿掉登入的可行性與風險

- **可行性：極高。** 登入只是 Splash 的一個本地 if 分支，改法都是幾行 smali 的活。
- **風險：低。** 不會讓 App 崩潰或擋住啟動；只有「需要帳號的雲端功能」會失效（且是溫和失效，不是崩）。純離線用 HUD＋地圖完全無礙。
- **不建議**：若勞大想「保留登入但改成可跳過」，請用下列方案 A 或 B，不要動 `UserController` 網路層（那會破壞真正需要的雲端功能）。

---

## 五、smali 層級具體改法建議

> 先 `apktool d MotoEye.apk` 解出 smali，改完 `apktool b` 再 `apksigner`/`bruce_moto.keystore` 重簽（目錄裡已有 keystore）。以下路徑以 apktool 產出的 smali 目錄為準。

### 方案 A（最推薦）：Splash 直接進主頁，恆真跳過登入
檔案：`smali/com/vidoar/motohud/view/SplashActivity.smali`

把 `init()` 中的分支改成「永遠走 msg=1（startMainPage）」。最乾淨的改法——把 `if-nez` 改成 `if-eqz`（恆真跳到主頁分支）：
```smali
# 原始：
#   invoke-static  {v0}, Landroid/text/TextUtils;->isEmpty(Ljava/lang/CharSequence;)Z
#   move-result v0
#   const-wide/16 v1, 0x5dc   # 1500
#   if-nez v0, :cond_0        # token 空才走登入
# 改為：
    if-eqz v0, :cond_0        # 任何情況都走 :cond_0（原登入分支）
```
或更暴力可靠：直接刪掉 `if-nez` 那一行（讓它 fall-through 到 `sendEmptyMessageDelayed(0,...)` 再強改 what=1）：
```smali
# 把
    if-nez v0, :cond_0
# 整行刪除，並把
    const/4 v3, 0x0          # sendEmptyMessageDelayed(0, ...)
# 改成
    const/4 v3, 0x1          # sendEmptyMessageDelayed(1, ...)  → 直接進主頁
```
> 兩種擇一即可。推薦「刪 `if-nez` + 把 what 改成 1」，最直觀，不會有 fall-through 歧義。

### 方案 B：把 startLoginPage 改去主頁（備援方案）
檔案：`smali/com/vidoar/motohud/view/SplashActivity.smali` 的 `startLoginPage()`：
```smali
# 原始：
.method startLoginPage()V
    new-instance v0, Landroid/content/Intent;
    const-class  v1, Lcom/vidoar/motohud/view/LoginActivity;
    invoke-direct {v0, v2, v1}, Landroid/content/Intent;-><init>(Landroid/content/Context;Ljava/lang/Class;)V
    invoke-virtual {v2, v0}, ...startActivity(Landroid/content/Intent;)V
    invoke-virtual {v2}, ...finish()V
    return-void
.end method
# 改為（只換 const-class）：
    const-class  v1, Lcom/vidoar/motohud/view/Main2Activity;
```
這樣就算別的地方（例如設定頁登出後）呼叫 `startLoginPage`，也會被導去主頁。

### 方案 C（保留登入但允許「遊客」）：讓 token 永遠非空
在 `InfoDataHelper.getAppToken()` 回傳值前塞一個假字串（需改 `SharedPreferences` 讀取或加預設），使 `isEmpty` 為 false。缺點：會讓雲端 API 帶著假 token 打伺服器、拿到 401 後某些功能報錯，**不如方案 A 乾淨**，不推薦。

### 重簽指令（目錄已有 keystore）
```bash
apktool b MotoEye_decoded -o MotoEye_nologin_unsigned.apk
# 用 /tmp 裡的 bruce_moto.keystore 重簽（別名/密碼需勞大提供，或自行 keytool 產生）
apksigner sign --ks bruce_moto.keystore --out MotoEye_nologin.apk MotoEye_nologin_unsigned.apk
```

## 六、驗證建議（改完要驗的）
1. 安裝後能直接進 `Main2Activity`，不彈登入頁 → 啟動分流成功。
2. 藍牙搜尋/連線 HUD、地圖顯示 → 核心功能正常（預期）。
3. 設定頁「登出」後不會被踢去登入（若採方案 B）→ 驗證導流。
4. 雲端 POI 收藏/帳號同步會失效 → 屬預期，不計入 bug。
