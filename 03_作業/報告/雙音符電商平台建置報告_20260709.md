## 一、專案定位

雙音符電商（DragonCommerce）是龍族的自有電商引擎平台。

核心商業模式：
- 我們是「平台方」，合作方（如面泌體 ExoMuse、未來咖啡豆）在平台上開店賣東西
- 錢「直接進合作方口袋」，我們不碰錢、不接 LINE Pay、不承擔支付合規風險
- 我們只統計「哪個店、哪個商品、透過我們網站賣出多少」
- 未來量大，按件 / 按比例跟合作方收平台費

## 二、目錄結構（模組化）

```
~/Desktop/龍族報告/01_專案/DragonCommerce/
├── core/
│   ├── tracker.py      ← 多店買賣數量追蹤核心（不含付款）
│   ├── server.py       ← 統計中心 API（port 8082）
│   └── sdk.js          ← DragonCommerce SDK（後門統計核心）
└── stores/
    ├── exosome/        ← 面泌體 ExoMuse（第一個合作店）
    │   ├── store.json   ← 店名、owner、api_key、payments
    │   └── products.json
    └── twotone/        ← TwoTone 龍族商城（直營測試店）
        ├── store.json
        └── products.json
```

每個合作方 = 一個獨立 store 模組。新增店只需複製 `stores/twotone/` 改 `store.json` + `products.json`，主程式零修改。

## 四、後門統計機制（關鍵）

目的：不管合作方把店部署在哪一個 GitHub（我們的 dragon-clan repo 或客戶自己新開的 repo），我們都能拿到賣出數據。

原理：
1. 合作方前端嵌入我們的 SDK：`<script src="https://<STATS_HOST>/sdk.js"></script>`
2. 顧客下單 → SDK 自動呼叫 `POST /track/sale`（帶 store_id + api_key）
3. 我們的統計中心寫入 `stores/<id>/sales_log.json` —— 數據所有權在我們手裡
4. 客戶前端只是展示層，沒有我們的 SDK 連不上下單邏輯，不會想拆掉

安全防護：`/track/sale` 強制 api_key 認證，錯誤 key 回 403。

驗證結果（2026-07-09）：
- 錯誤 api_key → 403 invalid api_key ✅
- 正確 api_key → ok, 數據寫入 ✅
- 總表即時反映賣出數量 ✅

## 六、結帳方式（適配器模式，預留接口）

平台不碰錢。每個 store 的 `store.json` 裡寫 `payments` 陣列，合作方自行選用：
- `qrcode` — 收款碼圖片（推薦先做，零 API、錢直達合作方）
- `manual` — 手動確認 / 貨到付款
- `linepay` — LINE Pay 真支付（合作方填自己的 Channel，未來接）
- `ecpay` — 綠界（合作方自己的，未來接）

目前 store.json 預設啟用 `qrcode` + `manual`。
